## A. 问题画像（固定——先记住它）

| 维度 | 本项目的值 |
|---|---|
| 变量类型 | 混合：连续（$w_i, \alpha$）+ 整数（窗口 $d_i$） |
| 维度范围 | 7 / 14 / 21（若结构搜索则更高且含离散） |

## B. 算法适配度清单（阅读时逐项打 ✓ / △ / ✗）

### B1. 搜索空间契合度
- [ ] 原生支持**连续**变量？
- [ ] 对**整数/混合**变量是否给出明确处理方式？（否则需自己加 rounding/embedding）
- [ ] 在**中低维（≤30）**上是否有 benchmark 报告？（警惕只在 100+ 维证明的算法）
- [ ] 对**非凸、多模态、阶跃**目标的鲁棒性证据？

### B2. 评估预算经济性
- [ ] 每次迭代所需的**评估次数** = 种群数 × 代数，是否与我们能负担的预算匹配？
- [ ] 是否支持**早停 / 预算敏感**的设计？
- [ ] 是否便于**并行化**（重要，能缩短实验时间）？

### B3. 方法学可信度（Q4/Q5 的延伸）
- [ ] Benchmark 选择是否合理？（CEC 系列 > 自造函数）
- [ ] Baseline 是否包含**非同家族**的经典算法（如 DE、CMA-ES、PSO）？
- [ ] 是否报告**均值+方差+统计检验**，而非只挑最好一次？
- [ ] 超参数是否**公平调**过（或用默认值）？
- [ ] 有无"no free lunch" 意识，还是声称"everywhere best"？

### B4. 可复现性（影响我们自己实现的成本）
- [ ] 伪代码是否**完整**（初始化、更新规则、终止条件齐全）？
- [ ] 是否给出**推荐超参数**及其敏感度分析？
- [ ] 有无官方/论文配套**参考实现**？（Project spec L386 允许改编）
- [ ] 关键公式中的符号是否定义清楚、无歧义？

### B5. 实现复杂度
- [ ] 更新规则可否在 **<100 行 Python** 内写完？
- [ ] 是否依赖**罕见数学工具**（如特定概率分布、图论结构）？
- [ ] 是否有**随机性来源过多**难以调试的风险？

### B6. 新颖性 vs 隐喻膨胀（Q3 的延伸）
- [ ] 核心机制能否**脱离自然隐喻**用数学讲清楚？（能——加分；只能讲故事——警惕）
- [ ] 与已有算法（PSO/DE/GA/ES）相比，**数学上**的真正差异是什么？
- [ ] 是否只是换了比喻没换本质？（这是该领域被广泛批评的现象）

### B7. 团队组合适配（Conclusions 用）
- [ ] 与队内其他算法**是否异质**？（同家族——扣分；跨 category——加分）
- [ ] 年代/谱系是否构成**可讲的故事**？（chronology/taxonomy）
- [ ] 与**单点基线**（爬山 / SA）互补度如何？

---

## C. 红旗（出现任一项 → 强烈建议换）

- ✗ 只在连续 benchmark 上验证，完全没讨论离散/混合。
- ✗ 论文只和**自家前作**比较，没和 PSO/DE/CMA-ES 这类"行业共识 baseline"比。
- ✗ 伪代码缺关键步骤，且没有任何公开代码。
- ✗ 超参数 ≥5 个且无敏感度分析。
- ✗ 新颖性完全建立在"自然隐喻新奇"上，数学上等价于已有算法。
- ✗ 实验只报"最好一次"的结果，没有多次运行统计。

---

## D. 绿旗（出现 → 加权推荐）

- ✓ 在 **CEC2017/2019/2020** 等标准 benchmark 上与多个异质 baseline 公平比较。
- ✓ 超参数少（≤3）或有**自适应**机制。
- ✓ 有官方代码且许可证允许改编。
- ✓ 原生或有成熟的**整数/混合变量**变体。
- ✓ 作者**坦诚**了局限（这种论文往往更可信）。

---

## 使用建议

- 每读完一篇论文，对 B1–B7 各子项打 ✓/△/✗ 并写一句证据引用（页码或段落）。
- 结果直接服务于该算法 synopsis 的 **Q6（你的评估）**。
- 全队选型确定后，可把各算法的画像对照表汇总到 `docs/comparison_summary.md` 作为 Conclusions 的论据。
